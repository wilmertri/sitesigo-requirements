# ADR-006: StaticPool para tests de integracion con SQLite

## Fecha
2025

## Estado
Aceptado

## Contexto
Los tests de integracion usan SQLite in-memory
(sqlite://) para no depender de una DB real.
El problema: SQLite in-memory crea una base de
datos distinta por cada conexion nueva.
Cuando FastAPI abre una sesion nueva por request
via Depends(get_db), las tablas creadas con
create_all() en el setup del test desaparecen.

## Decision
Usar StaticPool de SQLAlchemy en los tests
de integracion para forzar una sola conexion
compartida durante toda la duracion del test.

## Implementacion
En tests/integration/conftest.py:
engine = create_engine(
    "sqlite://",
    connect_args={"check_same_thread": False},
    poolclass=StaticPool
)

## Razon
StaticPool mantiene una unica conexion activa
y la reutiliza en todas las llamadas. Las tablas
creadas en el setup persisten durante el test
completo, incluyendo las conexiones que abre
FastAPI internamente.

## Consecuencias
- Tests de integracion completamente aislados
- Sin dependencia de archivos .db en disco
- El patron no aplica a produccion donde se
  usa PostgreSQL con connection pool real
- Cada test trabaja con una DB limpia si se
  recrea el engine en el fixture
