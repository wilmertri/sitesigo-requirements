# ADR-009: Alembic para migraciones de base de datos

## Estado
Aceptado

## Contexto
Al agregar soporte multi-tenant (proyectos), la tabla `requerimientos`
necesitaba una nueva columna `proyecto_id` y dos tablas nuevas.
El enfoque anterior (solo `Base.metadata.create_all`) crea tablas
que no existen pero no modifica tablas existentes en produccion.

Para el primer cambio se uso un script manual (`scripts/migrate_001.py`)
con `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`. Funciona, pero no es
sostenible para migraciones futuras.

## Decision
Usar **Alembic** como herramienta de migraciones a partir de ahora.

- `alembic revision --autogenerate -m "descripcion"` genera el diff
  entre los modelos SQLAlchemy y el estado real de la DB
- `alembic upgrade head` aplica todas las migraciones pendientes
- Las versiones quedan en `alembic/versions/` y se versionan en git

## Consecuencias

### Positivas
- Migraciones reproducibles y auditables
- `--autogenerate` detecta la mayoria de cambios automaticamente
- Compatible con PostgreSQL (produccion) y SQLite (desarrollo)
- Railway puede ejecutar `alembic upgrade head` como Release Command

### A tener en cuenta
- Los scripts manuales anteriores (migrate_001.py) deben ejecutarse
  una vez antes de iniciar el ciclo de migraciones con Alembic
- `--autogenerate` no detecta todos los cambios (ej. columnas renombradas,
  constraints complejos): revisar el diff antes de aplicar
- Para Railway: agregar en Settings → Deploy → Release Command:
  `alembic upgrade head`

## Comandos utiles

```bash
# Generar nueva migracion desde los modelos
alembic revision --autogenerate -m "descripcion del cambio"

# Aplicar todas las migraciones pendientes
alembic upgrade head

# Ver estado actual
alembic current

# Ver historial
alembic history
```
