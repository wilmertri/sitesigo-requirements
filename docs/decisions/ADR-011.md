# ADR-010: Coexistencia del enum base con estados configurables por proyecto

## Estado
Aceptado — 2026-06-08

## Contexto
ReqFlow tiene un ciclo de vida de estados fijo implementado
como enum EstadoRequerimiento:
  Nuevo | En analisis | En desarrollo | Resuelto |
  Cerrado | Rechazado | Archivado

La nueva funcionalidad (RN-23, RN-24) requiere que cada
proyecto defina su propia lista de estados en
ProyectoConfigEstado, con nombre, color y es_terminal.

Esto genera un conflicto: si los estados son configurables
por proyecto, ¿el enum base sigue teniendo sentido?
¿Hay que migrar el campo estado de todos los requerimientos?

Opciones evaluadas:
A) Reemplazar el enum por estados configurables completamente.
B) Conservar el enum y agregar estado_proyecto como campo
   separado en RequerimientooDB.

## Decision
Camino B: conservar el enum EstadoRequerimiento intacto
y agregar el campo estado_proyecto (String, nullable)
en la tabla requerimientos.

- El campo estado existente sigue representando el ciclo
  base de ReqFlow (RN-02, RN-07).
- El campo estado_proyecto representa el estado segun
  la configuracion del proyecto al que pertenece el
  requerimiento (RN-23, RN-24, RN-25).
- Ambos campos coexisten y son independientes.
- La logica de is_terminal para estado_proyecto vive en
  ConfigService, que consulta ProyectoConfigEstado.
- La logica de terminal para el estado base (RN-07)
  permanece en RequirementService sin cambios.

## Consecuencias
Positivas:
- Cero riesgo de romper el comportamiento en produccion.
  Los 52 tests existentes siguen en verde sin tocarlos.
- La migracion Alembic solo agrega una columna nullable;
  no modifica datos existentes.
- Separacion clara de responsabilidades: ciclo base vs
  flujo de proyecto.

A tener en cuenta:
- El frontend debera decidir cual estado mostrar primero
  (estado base o estado_proyecto). Eso es una decision
  de UX, no de backend.
- Al cambiar estado_proyecto, el historial (CambioEstado)
  debe registrar el cambio igual que con el estado base.
  Esto requiere extender cambiar_estado() o crear un
  metodo cambiar_estado_proyecto() en RequirementService.
- Un requerimiento puede tener estado=Cerrado (base) y
  estado_proyecto=En Revision (proyecto). Esta situacion
  es valida y refleja que el ciclo base y el flujo de
  proyecto son independientes.

## Alternativas descartadas
- Camino A (reemplazar el enum): descartado porque
  requiere migrar datos de produccion, reescribir
  RequirementService, cambiar todos los tests existentes
  y modificar el contrato HTTP. El riesgo supera el beneficio
  para el alcance actual.
