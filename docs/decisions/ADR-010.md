# ADR-009: Tablas proyectos y usuarios_proyectos son preexistentes

## Estado
Aceptado — 2026-06-08

## Contexto
Los modelos ProyectoConfigCampo, ProyectoConfigEstado y
RequerimientoValorCampo requieren un proyecto_id FK para
asociar configuraciones a un proyecto especifico.

Durante la revision arquitectonica surgio la pregunta:
¿existe ya una tabla proyectos en la BD, o proyecto_id
es un literal fijo para SITESIGO?

## Decision
Las tablas proyectos y usuarios_proyectos ya existen
en la base de datos de produccion. No se crean en
esta migracion.

La migracion Alembic de campos configurables solo
crea las 3 tablas nuevas y agrega las FK correspondientes
apuntando a proyectos.id.

## Consecuencias
Positivas:
- La migracion es mas pequeña y enfocada.
- No hay riesgo de duplicar entidades existentes.
- proyecto_id es un FK real con integridad referencial
  desde el primer dia.

A tener en cuenta:
- Antes de ejecutar la migracion, verificar que la tabla
  proyectos tenga al menos el registro de SITESIGO.
- El seed de SITESIGO (campos y estados iniciales) debe
  usar el id real del proyecto en BD, no un literal.
- Si en el futuro se agrega la tabla proyectos a los modelos
  SQLAlchemy del codigo (para relaciones ORM), se debe
  hacer con reflect o declarando el modelo sin recrear la tabla.

## Alternativas descartadas
- Crear tabla proyectos desde cero: descartado porque
  ya existe y recrearla romperia datos de produccion.
- Usar proyecto_id como string literal hardcodeado:
  descartado porque rompe integridad referencial y
  dificulta escalar a mas proyectos.
