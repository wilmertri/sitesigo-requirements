# ADR-004: Resolucion de import circular entre modelos

## Fecha
2026

## Estado
Aceptado

## Contexto
requirement.py necesita referenciar CambioEstado
(definido en history.py) para el campo historial.
history.py necesita referenciar EstadoRequerimiento
y RolUsuario (definidos en requirement.py).
Esto genera un import circular en runtime.

## Decision
Usar TYPE_CHECKING con from __future__ import annotations
en requirement.py para el import de CambioEstado.

## Razon
TYPE_CHECKING es True solo durante el analisis
estatico (mypy, pyright), nunca en runtime.
El import circular desaparece en ejecucion porque
el tipo solo existe como anotacion evaluada
de forma diferida gracias a annotations.
El service importa history.py normalmente porque
ningun modelo lo importa a el.

## Consecuencias
- Cero impacto en runtime
- Los type checkers siguen viendo los tipos correctos
- Patron estandar en Python para dependencias
  circulares entre modelos de dominio
